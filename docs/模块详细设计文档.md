## 3. 入口模块：`main.py`

## 3.1 模块职责

`main.py` 是 AstrBot 插件入口，负责：

- 注册插件
- 绑定 AstrBot 事件与命令
- 初始化核心对象
- 处理私聊与群聊分流
- 控制插件关闭流程

它已经被重构为“**薄路由层**”，业务逻辑尽量下沉到 `core/`、`handlers/`、`services/`。

---

## 3.2 核心类：`EngramPlugin`

### 初始化流程

在 `__init__()` 中完成：

1. 读取并兼容 AstrBot 配置
2. 通过 `ConfigPresetService` 应用配置预设
3. 获取插件数据目录
4. 初始化 `MemoryFacade`
5. 初始化：
   - `ExportHandler`
   - `ProfileRenderer`
   - `MemoryCommandHandler`
   - `ProfileCommandHandler`
   - `OneBotSyncHandler`
   - `MemoryToolHandler`
6. 初始化服务：
   - `LLMContextInjector`
   - `IntentClassifier`
   - `TopicMemoryCacheService`
   - `ToolHintStrategyService`
   - `TimeExpressionService`
   - `FriendCacheService`
7. 启动 `MemoryScheduler`
8. 按配置选择是否启动 `EngramWebServer`

## 3.4 事件入口设计

### 3.4.1 私聊入口：`on_private_message`

职责：

- 过滤命令消息
- 调用 `logic.record_message()` 写入原始消息
- 调用 `OneBotSyncHandler.sync_user_info()` 同步用户基础资料

### 3.4.2 LLM 请求前入口：`on_llm_request`

职责：

- 获取用户画像
- 判断是否需要检索长期记忆
- 命中记忆时构建 `【长期记忆回溯】`
- 根据策略决定是否加工具提示
- 通过 `LLMContextInjector` 注入 system prompt

### 3.4.3 群聊请求前入口：`_handle_group_llm_request`

职责：

- 检查群聊记忆开关与好友白名单
- 过滤噪声和指令消息
- 在群聊场景执行记忆检索与画像注入
- 暂存待写入的群聊消息上下文

### 3.4.4 回复后入口：`after_message_sent`

职责：

- 记录 assistant 回复到原始消息池
- 更新互动统计

### 3.4.5 群聊回复后入口：`_handle_group_after_message_sent`

职责：

- 在群聊 LLM 回复完成后，把用户消息和 assistant 回复写入群聊原始库

## 3.5 命令路由设计

`main.py` 中命令函数本质上只做两件事：

1. 参数校验/转发
2. 调用 handler/core 返回结果

例如：

- `/mem_list` → `MemoryCommandHandler.handle_mem_list`
- `/profile show` → `ProfileCommandHandler.handle_profile_show`
- `/mem_export` → `ExportHandler.handle_export_command`

这样可让主文件保持简洁，逻辑集中在专门模块中。

## 5.3 子职责拆解

## 5.3.1 向量管理

关键方法：

- `_ensure_chroma_initialized()`
- `_get_embedding_provider()`
- `_ensure_embeddings()`
- `_collection_add_texts()`
- `_collection_query_text()`

设计特点：

- 延迟初始化 ChromaDB
- 强制使用配置的 embedding provider
- 支持 provider 返回格式归一化
- 自动识别 batch size 上限并分片
- 识别维度不匹配错误并给出重建提示

### 设计意图

向量是增强能力，不是主链路唯一依赖。因此：

- embedding 不可用时，记录警告
- 写向量失败不阻塞 SQLite 主流程
- 检索失败时支持回退关键词检索

---

## 5.3.2 原始消息记录

方法：`record_message()`

处理逻辑：

1. 过滤无效内容
2. 生成消息 UUID
3. 写入 `RawMemory`
4. 若角色是 `user`：
   - 更新最后聊天时间
   - 更新未归档消息计数

### 过滤逻辑

通过 `_is_valid_message_content()` 实现：

- 过滤命令消息
- 过滤内部短指令
- 过滤过短无意义文本

---

## 5.3.3 归档总结

主入口：

- `check_and_summarize()`
- `_summarize_private_chat(user_id)`

### 归档触发条件

- 私聊：超过 `private_memory_timeout`
- 私聊：`unsaved_msg_count >= min_msg_count`
- 群聊：优先使用 `group_memory_timeout`
- 群聊：优先使用 `group_min_msg_count`
- 若群聊专用配置未填写，则分别回退到私聊配置

### 归档流程

1. 查询用户所有未归档原始消息
2. 按日期分组
3. 每组调用 `_process_single_summary_batch()`
4. 生成长期记忆摘要
5. 先写 `MemoryIndex` 到 SQLite
6. 再把相关原始消息标记为已归档
7. 最后尝试写入 ChromaDB
8. 如果写向量失败，加入待补偿队列

### 设计亮点

- **先落 SQLite，后写向量**，避免 embedding 问题导致记忆丢失
- 使用 `prev_index_id` 把记忆连接成时间链
- 使用 `ref_uuids` 保存原始消息指针

---

## 5.3.4 单批次总结处理

方法：`_process_single_summary_batch()`

职责：

- 过滤当日无效消息
- 组装聊天文本
- 套用 `summarize_prompt`
- 调 LLM 总结
- 解析结构化 JSON 输出
- 构造适合检索的摘要文本

### 结构化摘要支持

若 LLM 返回 JSON，会提取：

- `summary`
- `key_facts`
- `keywords`
- `entities`
- `mood`

并通过 `_build_structured_summary()` 拼接为更适合向量检索的文本。

---

## 5.3.5 检索与排序

主入口：`retrieve_memories()`

### 检索阶段

1. 意图分类
2. 构造 Chroma where 过滤
3. 向量召回候选集
4. 过滤低相关性结果
5. 计算排序特征
6. 执行 `rrf` 或 `hybrid` 排序
7. 补充前情时间线与原文预览
8. 对命中记忆执行活跃度强化

### 意图分类影响

根据 `IntentClassifier.classify_query()` 得到：

- `skip`
- `recall`
- `preference_fact`
- `event_narrative`

这些类别会影响：

- 相似度阈值
- 关键词/向量权重
- 时效分比重

### 排序特征

- `vector_score`
- `keyword_score_norm`
- `recency_score`
- `activity_score`

### 展示增强

每条结果会附带：

- 短 ID
- 时间
- 摘要
- 前情时间线
- 原文预览
- 相关性百分比

---

## 5.3.6 关键词回退检索

方法：`_retrieve_memories_by_keyword_fallback()`

触发条件：

- Chroma 查询失败
- 向量为空
- embedding 不可用

处理逻辑：

1. 从 SQLite 中按关键词召回候选
2. 本地做关键词重排
3. 同样补充时间线与原文预览
4. 同样提升命中记忆的 `active_score`

### 设计价值

即使向量不可用，项目仍保有“可搜索”的基础能力。

---

## 5.3.7 删除与撤销

删除相关方法：

- `_delete_memory_entry()`
- `delete_memory_by_sequence()`
- `delete_memory_by_id()`

撤销相关方法：

- `undo_last_delete()`

### 删除逻辑

1. 备份索引信息
2. 尝试备份向量数据
3. 记录到 `_delete_history`
4. 从 Chroma 删除向量
5. 如果 `delete_raw=True`，删除原始消息
6. 否则把原始消息标为未归档
7. 从 SQLite 删除 MemoryIndex

### 撤销逻辑

1. 取最近删除记录
2. 恢复 SQLite MemoryIndex
3. 恢复向量数据
4. 恢复原始消息的归档状态

---

## 5.3.8 多级折叠总结

核心方法：`_fold_summaries()`

对外方法：

- `fold_weekly_summaries()`
- `fold_monthly_summaries()`
- `fold_yearly_summaries()`

### 通用流程

1. 根据 lower-level source_type 查询候选摘要
2. 若时间窗口内为空，回退到最近记忆列表
3. 不足最小样本则跳过
4. 按时间正序拼接摘要列表
5. 调用 LLM 生成更高层总结
6. 写入 SQLite MemoryIndex
7. 尝试写入向量库
8. 向量失败则进入待补偿队列

---

## 5.3.9 向量重建

方法：`rebuild_vector_collection()`

### 功能

- 从 SQLite 全量读取 `MemoryIndex`
- 重新写入 ChromaDB

### full_rebuild 模式

- 备份旧 Chroma 目录
- 删除旧集合
- 创建新集合
- 按批次重建全部向量

适用于：

- embedding_provider 切换
- 向量维度不匹配
- 补偿失败后人工重建

## 7. 调度模块：`core/scheduler.py`

## 7.1 模块职责

统一管理后台任务：

- 归档检测
- 每日画像更新
- 每日记忆维护
- 周/月/年折叠总结

---

## 7.2 核心设计

### 状态与可观测性

- `_tasks`：保存任务引用，便于关闭时统一取消
- `_task_metrics`：保存运行次数、成功失败、耗时、错误信息、跳过原因

### 中断安全

所有任务在执行前都会检查：

- `self._is_shutdown`
- `logic._is_shutdown`
- `executor._shutdown`

---

## 7.3 主要任务

### 7.3.1 `background_worker()`

负责归档检测。
特点：动态计算下一次检测时间，而不是固定死循环轮询。

补充说明：
- 私聊链路默认使用 `private_memory_timeout` 与 `min_msg_count`
- 群聊链路若启用，则优先使用 `group_memory_timeout` 与 `group_min_msg_count`
- 群聊未配置时回退到私聊配置

### 7.3.2 `daily_persona_scheduler()`

每天 00:00 执行，更新的是“昨天”的画像。
支持：

- 并发控制
- 每用户延迟
- 活跃用户遍历

### 7.3.3 `daily_memory_maintenance()`

每天 01:00 执行：

- 记忆衰减
- 冷记忆修剪

### 7.3.4 `weekly_folding_scheduler()`

根据配置的星期与小时执行周折叠。

### 7.3.5 `monthly_folding_scheduler()`

按月执行月折叠。

### 7.3.6 `yearly_folding_scheduler()`

按年执行年折叠。

## 7.4 活动记录设计

调度器会通过 `_push_activity()` 把任务完成情况推入 MemoryManager 的近期动态列表，供 WebUI 首页展示。

## 9.3 `DatabaseManager`

## 16. 模块间调用关系总结

### 16.1 私聊链路

`main.py`
→ `MemoryFacade.record_message()`
→ `MemoryManager.record_message()`
→ `DatabaseManager.save_raw_memory()`

并行：

`main.py`
→ `OneBotSyncHandler.sync_user_info()`
→ `ProfileManager.update_user_profile()`

### 16.2 LLM 注入链路

`main.py.on_llm_request`
→ `ProfileManager.get_user_profile()`
→ `IntentClassifier.should_retrieve_memory()`
→ `MemoryManager.retrieve_memories()`
→ `LLMContextInjector.inject_context()`

### 16.3 归档链路

`MemoryScheduler.background_worker()`
→ `MemoryFacade.check_and_summarize()`
→ `MemoryManager._summarize_private_chat()`
→ SQLite `MemoryIndex`
→ ChromaDB

### 16.4 画像更新链路

`MemoryScheduler.daily_persona_scheduler()`
→ `ProfileManager.update_persona_daily()`
→ LLM
→ `ProfileGuardian.validate_update()`
→ 快照
→ JSON 落盘

### 16.5 WebUI 链路

`webui/static/*.js`
→ `webui_server.py API`
→ `MemoryFacade / ProfileManager / MemoryManager / DatabaseManager`
