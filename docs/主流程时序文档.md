# astrbot_plugin_engram 主流程时序文档

## 1. 文档目的

本文档描述 `astrbot_plugin_engram` 的核心运行时流程，重点说明：

- 消息是如何进入记忆系统的
- 长期记忆是如何归档、检索、注入的
- 用户画像是如何更新的
- 群聊链路与私聊链路有何差异
- 删除、撤销、导出、WebUI 的主要时序关系

项目路径：`E:/AI/shouban/astrbot_plugin_engram`

---

## 2. 核心参与者

为便于理解，下文统一使用以下参与者名称：

| 参与者 | 说明 |
|---|---|
| 用户 | 最终聊天对象 |
| AstrBot | 宿主框架，触发消息事件与 LLM 请求 |
| `main.py / EngramPlugin` | 插件入口与路由层 |
| `MemoryFacade` | 记忆与画像统一门面 |
| `MemoryManager` | 原始消息、归档、检索、删除、向量管理核心 |
| `ProfileManager` | 画像读写、每日更新、快照、回滚核心 |
| `DatabaseManager` | SQLite 数据访问层 |
| ChromaDB | 长期记忆向量检索层 |
| LLM Provider | 总结、画像更新、对话生成依赖的大模型 |
| `MemoryScheduler` | 后台归档、画像、维护调度器 |
| WebUI Server | 管理后台 API 层 |

## 3. 系统启动流程

## 3.1 流程说明

插件加载时，`EngramPlugin.__init__()` 会完成配置合并、核心对象初始化、调度器启动，以及可选的 WebUI 启动。

## 3.2 时序图

```mermaid
sequenceDiagram
    participant AstrBot
    participant Plugin as EngramPlugin
    participant Facade as MemoryFacade
    participant DB as DatabaseManager
    participant PM as ProfileManager
    participant MM as MemoryManager
    participant Scheduler as MemoryScheduler
    participant WebUI as EngramWebServer

    AstrBot->>Plugin: 创建插件实例
    Plugin->>Plugin: 读取并应用配置预设
    Plugin->>Facade: 初始化 MemoryFacade(data_dir)
    Facade->>DB: 初始化 SQLite / 校验契约
    Facade->>PM: 初始化画像管理器
    Facade->>MM: 初始化记忆管理器
    Plugin->>Plugin: 初始化 handlers / services
    Plugin->>Scheduler: 创建并启动调度器
    alt 启用 WebUI
        Plugin->>WebUI: 创建并启动服务
    end
```

## 3.3 结果

系统启动后会形成以下长期运行能力：

- 私聊消息记录
- LLM 请求前记忆检索与 prompt 注入
- 定时归档总结
- 定时画像更新
- 记忆维护与折叠总结
- 可选 WebUI 管理端

## 4. 私聊消息录入流程

## 4.1 场景说明

当用户发送私聊消息时，插件会优先做“原始消息记录”，但不会立刻生成长期记忆。长期记忆由后台调度器在满足条件后异步归档。

## 4.2 时序图

```mermaid
sequenceDiagram
    participant User as 用户
    participant AstrBot
    participant Plugin as EngramPlugin.on_private_message
    participant Facade as MemoryFacade
    participant MM as MemoryManager
    participant DB as DatabaseManager
    participant Sync as OneBotSyncHandler
    participant PM as ProfileManager

    User->>AstrBot: 发送私聊消息
    AstrBot->>Plugin: on_private_message(event)
    Plugin->>Plugin: 判断是否为命令消息
    alt 普通消息
        Plugin->>Facade: record_message(user_id, session_id, role=user, content)
        Facade->>MM: record_message(...)
        MM->>DB: save_raw_memory(...)
        MM->>MM: 更新 last_chat_time / unsaved_msg_count
        Plugin->>Sync: sync_user_info(event)
        Sync->>PM: update_user_profile(基础资料增量)
    else 指令消息
        Plugin-->>AstrBot: 跳过记忆写入
    end
```

## 4.3 关键点

1. **先写原始消息，不立即总结**
2. **命令消息默认不进入记忆**
3. **用户资料同步与消息记录并行发生**
4. **原始消息是后续归档、导出、回溯的唯一事实源**

## 5. 私聊回复后记录流程

## 5.1 场景说明

当 AstrBot 完成回复后，插件会把 assistant 回复也写入原始消息表，并更新画像中的互动统计。

## 5.2 时序图

```mermaid
sequenceDiagram
    participant AstrBot
    participant Plugin as EngramPlugin.after_message_sent
    participant Facade as MemoryFacade
    participant MM as MemoryManager
    participant DB as DatabaseManager
    participant PM as ProfileManager

    AstrBot->>Plugin: after_message_sent(event, result)
    Plugin->>Facade: record_message(role=assistant, content=reply)
    Facade->>MM: record_message(...)
    MM->>DB: save_raw_memory(...)
    Plugin->>PM: update_interaction_stats(user_id)
    PM->>PM: 更新 first_chat_date / last_chat_date / total_chat_days / total_valid_chats
    PM->>文件: 写回 engram_personas/{user_id}.json
```

## 5.3 关键点

- 记忆系统保存的是双边对话，而非只有用户单边输入
- 画像中的互动统计在回复完成后更新，更接近一次完整交互闭环

## 6. LLM 请求前记忆检索与注入流程

## 6.1 场景说明

当用户发起新问题，AstrBot 即将调用大模型前，插件会判断是否需要检索长期记忆，并将画像与记忆回溯块注入到 prompt 中。

## 6.2 时序图

```mermaid
sequenceDiagram
    participant User as 用户
    participant AstrBot
    participant Plugin as EngramPlugin.on_llm_request
    participant PM as ProfileManager
    participant IC as IntentClassifier
    participant MM as MemoryManager
    participant Chroma as ChromaDB
    participant DB as DatabaseManager
    participant Injector as LLMContextInjector
    participant LLM as LLM Provider

    User->>AstrBot: 提问
    AstrBot->>Plugin: on_llm_request(event)
    Plugin->>PM: get_user_profile(user_id)
    Plugin->>IC: should_retrieve_memory(query)
    alt 需要检索
        Plugin->>MM: retrieve_memories(user_id, query)
        MM->>Chroma: query(语义召回)
        Chroma-->>MM: 候选 index_id / metadata / document
        MM->>DB: 批量回查 MemoryIndex
        MM->>DB: 批量回查 RawMemory(ref_uuids)
        MM->>DB: update_active_score(命中记忆)
        MM-->>Plugin: 结构化回溯文本
    else 跳过检索
        Plugin-->>Plugin: 仅保留画像或当前上下文
    end
    Plugin->>Injector: inject_context(profile_block, memory_block)
    Injector-->>AstrBot: 注入后的 prompt
    AstrBot->>LLM: 发起正式对话请求
```

## 6.3 关键点

### 6.3.1 检索不是每次都执行

先由 `IntentClassifier` 判断消息是否值得检索，避免无意义查询浪费成本。

### 6.3.2 检索链路是“向量召回 + SQLite 回查”

Chroma 负责找候选，SQLite 负责补全：

- 摘要正文
- 前序链路 `prev_index_id`
- 原文列表 `ref_uuids`
- 活跃度加权

### 6.3.3 向量失败可降级

若 Chroma 不可用或 embedding 失败，会降级到 SQLite 关键词检索。

## 7. 长期记忆归档流程

## 7.1 场景说明

长期记忆不是实时生成，而是由后台调度器周期性检查。当满足“超时 + 条数阈值”后，系统才对未归档原始消息进行总结。

## 7.2 时序图

```mermaid
sequenceDiagram
    participant Scheduler as MemoryScheduler
    participant Facade as MemoryFacade
    participant MM as MemoryManager
    participant DB as DatabaseManager
    participant LLM as LLM Provider
    participant Chroma as ChromaDB

    Scheduler->>Facade: check_and_summarize()
    Facade->>MM: check_and_summarize()
    MM->>MM: 遍历活跃用户，检查 timeout / min_msg_count
    MM->>DB: get_unarchived_raw(session_id)
    DB-->>MM: 未归档原始消息
    MM->>MM: 按日期分组
    loop 每个日期批次
        MM->>LLM: 调用总结 Prompt
        LLM-->>MM: summary / 结构化 JSON
        MM->>DB: save_memory_index(...)
        MM->>DB: mark_as_archived(ref_uuids)
        MM->>Chroma: add(ids, documents, metadatas)
        alt 向量写入失败
            MM->>MM: 记录 pending_vector_jobs
        end
    end
```

## 7.3 触发条件

典型条件：

- 距最后一次用户消息超过 `private_memory_timeout`
- 未归档消息数达到 `min_msg_count`

## 7.4 关键点

1. **按天分组总结**：避免跨日上下文过长、时间粒度混乱
2. **先写 SQLite，再写 Chroma**：主数据优先落地
3. **原始消息标记归档而非立刻删除**：便于回溯和重建
4. **新记忆通过 `prev_index_id` 串成时间链**

## 8. 画像每日更新流程

## 8.1 场景说明

每天由调度器对符合条件的用户进行画像深度更新，使用当天或指定时间范围的长期记忆作为输入。

## 8.2 时序图

```mermaid
sequenceDiagram
    participant Scheduler as MemoryScheduler
    participant PM as ProfileManager
    participant DB as DatabaseManager
    participant LLM as LLM Provider
    participant Guardian as ProfileGuardian
    participant Files as Persona JSON Files

    Scheduler->>PM: update_persona_daily(user_id, start_time, end_time)
    PM->>DB: get_memories_in_range(...) / get_memories_since(...)
    DB-->>PM: 当日 MemoryIndex 列表
    PM->>Files: 读取当前画像
    PM->>LLM: persona_update_prompt(current_persona + memory_texts)
    LLM-->>PM: proposal JSON
    PM->>Guardian: validate_update(current, proposal, evidence)
    Guardian-->>PM: validated_persona / conflicts / decisions
    PM->>PM: merge _meta / 衰减 stale preferences
    PM->>Files: snapshot 旧画像
    PM->>Files: 覆盖写入新画像
```

## 8.3 关键点

- 画像不是直接相信 LLM 输出，而是先经过 `ProfileGuardian`
- 写入前先做快照，支持后续回滚
- `_meta` 中会累计 accepted field 的证据链
- `likes/dislikes` 支持 TTL 衰减，防止画像老化

## 9. 群聊记忆流程

## 9.1 场景说明

群聊链路与私聊链路类似，但它是**按需延迟初始化**的，并使用独立 SQLite 与 Chroma 存储。

## 9.2 初始化时序图

```mermaid
sequenceDiagram
    participant Plugin as EngramPlugin
    participant DB as DatabaseManager
    participant MM as MemoryManager
    participant Scheduler as MemoryScheduler

    Plugin->>Plugin: _ensure_group_memory_manager()
    Plugin->>DB: 初始化 engram_memories_group.db
    Plugin->>MM: 初始化群聊 MemoryManager(chroma_group, default_source_type=group)
    Plugin->>Scheduler: 启动群聊调度器
```

## 9.3 请求前注入时序图

```mermaid
sequenceDiagram
    participant GroupUser as 群成员
    participant AstrBot
    participant Plugin as EngramPlugin._handle_group_llm_request
    participant GroupMM as Group MemoryManager
    participant PM as ProfileManager

    GroupUser->>AstrBot: 群聊提问
    AstrBot->>Plugin: 群聊 LLM 请求前事件
    Plugin->>Plugin: 检查群聊记忆开关 / 白名单 / 最小字数
    Plugin->>Plugin: 解析 storage_id(group_id 或 user_id)
    Plugin->>GroupMM: retrieve_memories(storage_id, query)
    Plugin->>PM: 可选读取私聊画像
    Plugin-->>AstrBot: 注入群聊记忆与画像提示
```

## 9.4 回复后记录时序图

```mermaid
sequenceDiagram
    participant AstrBot
    participant Plugin as EngramPlugin._handle_group_after_message_sent
    participant GroupMM as Group MemoryManager
    participant GroupDB as Group DatabaseManager

    AstrBot->>Plugin: 群聊回复完成
    Plugin->>GroupMM: record_message(role=user, ...)
    GroupMM->>GroupDB: save_raw_memory(用户消息)
    Plugin->>GroupMM: record_message(role=assistant, ...)
    GroupMM->>GroupDB: save_raw_memory(助手回复)
```

## 9.5 群聊与私聊的关键差异

| 项目 | 私聊 | 群聊 |
|---|---|---|
| 数据库 | `engram_memories.db` | `engram_memories_group.db` |
| Chroma 路径 | `engram_chroma/` | `engram_chroma_group/` |
| 初始化方式 | 插件启动即初始化 | 首次需要时延迟初始化 |
| 画像 | 完整支持 | 主要复用私聊画像，不单独维护群画像 |
| 存储归属 | `user_id` | 可配置为 `group_id` 或 `user_id` |

## 10. 删除与撤销流程

## 10.1 删除流程

```mermaid
sequenceDiagram
    participant User as 用户/管理员
    participant Handler as MemoryCommandHandler
    participant MM as MemoryManager
    participant DB as DatabaseManager
    participant Chroma as ChromaDB

    User->>Handler: /mem_delete 或 /mem_delete_all
    Handler->>MM: delete_memory_by_sequence / delete_memory_by_id
    MM->>DB: 查询目标 MemoryIndex
    MM->>Chroma: 读取向量备份（可选）
    MM->>MM: 写入 _delete_history
    MM->>Chroma: delete(ids=[index_id])
    alt delete_raw = true
        MM->>DB: delete_raw_memories_by_uuids(ref_uuids)
    else delete_raw = false
        MM->>DB: RawMemory.update(is_archived=false)
    end
    MM->>DB: delete_memory_index(index_id)
```

## 10.2 撤销流程

```mermaid
sequenceDiagram
    participant User as 用户/管理员
    participant Handler as MemoryCommandHandler
    participant MM as MemoryManager
    participant DB as DatabaseManager
    participant Chroma as ChromaDB

    User->>Handler: /mem_undo
    Handler->>MM: undo_last_delete(user_id)
    MM->>MM: 读取 _delete_history 最近记录
    MM->>DB: save_memory_index(恢复 MemoryIndex)
    alt 有备份向量
        MM->>Chroma: add(恢复向量)
    else 无备份向量
        MM->>Chroma: 重新生成 summary 向量
    end
    MM->>DB: RawMemory.update(is_archived=true)
```

## 10.3 关键点

- 撤销历史保存在内存中，不跨重启
- 删除支持“只删索引”与“索引 + 原文一起删”两种模式
- 若保留原文，则删除长期记忆时会把原始消息重新置为未归档，便于后续再次总结

## 12. WebUI 查询流程

## 12.1 场景说明

WebUI 通过 `FastAPI + 线程池` 访问 SQLite、画像文件和渲染器，提供统计、记忆浏览、画像管理等能力。

## 12.2 时序图

```mermaid
sequenceDiagram
    participant Browser as 浏览器
    participant API as EngramWebServer
    participant Facade as MemoryFacade/Manager
    participant DB as DatabaseManager
    participant PM as ProfileManager
    participant Renderer as ProfileRenderer

    Browser->>API: GET /api/stats/overview
    API->>DB: 统计消息、用户、归档量
    API-->>Browser: 返回统计 JSON

    Browser->>API: GET /api/memories
    API->>Facade: 查询记忆列表/详情
    Facade->>DB: 读取 MemoryIndex 与 RawMemory
    API-->>Browser: 返回记忆数据

    Browser->>API: GET /api/profile/{user_id}
    API->>PM: get_user_profile(user_id)
    API-->>Browser: 返回画像 JSON

    Browser->>API: GET /api/profile/{user_id}/render
    API->>Renderer: render(profile)
    API-->>Browser: 返回渲染图片
```

## 12.3 关键点

- 阻塞型数据库操作通过线程池执行
- 私聊与群聊统计需要分别读取再汇总
- WebUI 当前更偏运维与浏览，不是事务型后台

## 13. 异常与降级流程

## 13.1 向量不可用降级

```mermaid
flowchart TD
    A[开始检索] --> B{embedding provider / Chroma 可用?}
    B -->|是| C[执行向量检索]
    B -->|否| D[执行 SQLite 关键词检索]
    C --> E{检索成功?}
    E -->|是| F[返回记忆结果]
    E -->|否| D
    D --> F
```

## 13.2 画像读取失败降级

- 若画像文件不存在：返回默认画像
- 若画像文件损坏：记录日志并回退默认画像
- 若历史文件损坏：回滚功能返回失败信息，但不影响主链路聊天

## 13.3 WebUI 降级

- 若 WebUI 启动失败，不影响主聊天链路
- 若群聊数据库未启用，统计页可只展示私聊数据

## 14. 一页式主流程总览

```mermaid
flowchart TD
    A[用户发消息] --> B{私聊/群聊?}
    B -->|私聊| C[记录 RawMemory]
    B -->|群聊| D[记录群 RawMemory]

    C --> E[LLM 请求前检索长期记忆]
    D --> F[群聊检索长期记忆]

    E --> G[注入画像 + 记忆块]
    F --> G
    G --> H[大模型生成回复]
    H --> I[记录 assistant 回复]

    I --> J[后台调度检查归档]
    J --> K[生成 summary]
    K --> L[写 MemoryIndex]
    L --> M[标记 RawMemory 已归档]
    M --> N[写 Chroma 向量]

    L --> O[后续检索/删除/导出/WebUI]
    O --> P[每日画像更新]
```

## 15. 一句话总结

`astrbot_plugin_engram` 的主链路可以概括为：

**先把对话完整保存为原始消息，再由后台异步沉淀成长期记忆，并在下一次对话前把“画像 + 记忆回溯”重新注入给 LLM。**
